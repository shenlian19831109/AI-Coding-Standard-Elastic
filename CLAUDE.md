# CLAUDE.md - AI 编程弹性约束规范 (V2.0)

本规范旨在通过 **MODE (Strict / Balanced / Fast)** 变量动态调整 AI 的行为边界，在确保代码质量与安全的同时，最大化开发效率。

---

## 🔧 模式定义与切换
用户可通过对话指令（如 "使用 strict 模式"）或项目配置文件（`.clauderc` 或环境变量 `AI_MODE`）切换模式。

| 维度 | **Fast (原型模式)** | **Balanced (日常模式)** | **Strict (审计模式)** |
| :--- | :--- | :--- | :--- |
| **核心目标** | 极速迭代、跑通原型 | 兼顾效率与质量（默认） | 零幻觉、高可靠、可审计 |
| **防幻觉** | 允许常识性推断 | 仅不熟悉库强制溯源 | **全量强制溯源** |
| **命名一致性** | 仅单次对话/同文件对齐 | 自动对齐项目风格/术语表 | **严格遵循术语表** |
| **询问频率** | 不询问，采用安全默认值 | 仅关键歧义询问 | **不确定即停机询问** |
| **证据链标注** | 不要求 | 不要求 | **关键 API/配置强制标注** |
| **自检方式** | 基础运行检查 | Lint + 类型检查 | **单元测试 + 严格类型** |

---

## 🛡️ 通用底线 (All Modes)
无论处于何种模式，以下规则均为不可逾越的红线：

1.  **禁止编造 API**：严禁使用未在上下文出现、未读取过文档、未在项目中存在的函数或字段。
2.  **禁止同义异名**：同一概念在单次任务中必须保持命名唯一性（如已用 `user_id`，严禁再用 `uid`）。
3.  **安全红线**：
    *   **SQL**：严禁字符串拼接，必须使用参数化查询。
    *   **Shell**：严禁 `os.system(user_input)`，必须使用数组传参的 `subprocess`。
    *   **权限**：严禁 `rm -rf /` 或 `chmod 777`（除非用户明确授权）。

---

## 🟢 Fast 模式 (适合 MVP / 脚本 / 样式调整)
*   **推断策略**：遇到不确定的 API 或逻辑，按最常见用法编写，并标注 `# TODO: verify`。
*   **优化建议**：允许主动重构或简化逻辑，无需事先询问。
*   **交互逻辑**：不打断用户，优先完成功能交付。

## 🟡 Balanced 模式 (建议日常业务开发使用)
*   **溯源要求**：调用不熟悉的第三方 SDK 前，必须使用 `Read` 或 `WebFetch` 读取文档。
*   **命名对齐**：自动检查项目根目录是否存在 `GLOSSARY.md`，若有则强制复用。
*   **询问逻辑**：仅在涉及业务逻辑冲突（如：订单状态流转歧义）时询问，实现细节由 AI 自行决策。
*   **优化逻辑**：发现更优方案时，在输出中简要说明理由，但默认按用户原始需求执行。

## 🔴 Strict 模式 (核心模块 / 金融支付 / SDK 开发)
*   **强制溯源**：任何非标准库调用前，必须提供文档来源或项目内调用示例。
*   **证据链标注**：
    *   首次调用的第三方 API 需标注：`# [来源] 库名 v版本 - 示例/章节`。
    *   复杂配置项需标注原因。
*   **不确定性处理**：信息不完整时立即停止编码，列出清单询问用户。
*   **质量约束**：代码必须包含类型注解（Type Hints / TS Strict），且关键逻辑必须附带单元测试。

---

## 📋 自动化自检 (替代 Checklist)
AI 在交付代码前，必须根据模式完成以下自动化约束：
1.  **Python**：确保通过 `mypy` 检查（Strict 模式）或具备基础类型提示（Balanced 模式）。
2.  **JS/TS**：确保符合项目 `ESLint` 规则，TS 严禁使用 `any`（Strict 模式）。
3.  **SQL**：检查所有查询是否已参数化。
4.  **命名**：检查是否与项目已有代码风格（Snake/Camel）一致。

---

## 📁 辅助资源加载
AI 会根据场景自动读取以下文件以增强能力：
*   `GLOSSARY.md`：业务术语与命名映射表。
*   `.claude/rules.md`：项目特定架构约束（如：所有时间必须使用 UTC）。
*   `.claude/skills/*.md`：特定技术栈（React, SQL, Security）的深度规范。
